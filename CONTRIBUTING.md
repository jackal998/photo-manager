# Contributing to photo-manager

Welcome. This is a Windows PySide6 desktop app for deduplication scanning
and review. The notes below are the bits that aren't obvious from the
code — read them once before your first PR.

For environment setup, run instructions, and the usage tour, see the
[README](README.md).

---

## Setup notes that aren't in the README

The README install steps cover the basics. A few things that are easy
to miss on a fresh checkout:

- **`exiftool` on `PATH` is required** — the scanner shells out for
  EXIF date extraction. Install via `choco install exiftool` (Windows)
  or download from [exiftool.org](https://exiftool.org/). Without it,
  the scanner crashes on the first non-trivial photo.
- **`settings.json` is gitignored.** Copy from `settings.json.example`
  before first run; the app tolerates a missing file but you won't get
  your source folders persisted.
- **`.claude/settings.json` is also gitignored** (only relevant if you
  use Claude Code). Copy from `.claude/settings.json.example` and
  replace `<USER_HOME>` with your actual home directory — the file
  carries the security-gate permissions config.
- **QA fixtures live in `qa/sandbox/`.** Most are checked in; a few
  large ones get auto-generated by `scripts/make_qa_sandbox.py` when
  the QA batch runs.

## PR template and pre-merge checklist

Every new PR auto-fills from
[`.github/PULL_REQUEST_TEMPLATE.md`](.github/PULL_REQUEST_TEMPLATE.md).
GitHub picks it up automatically regardless of how the PR is opened
(CLI, web UI, mobile, fork) — you don't need to copy it in by hand.

The template puts the **Pre-merge checklist** above **Summary** on
purpose: reviewers scanning a PR see the doc / test / qa-scenario
boxes first, which makes drift cheap to catch even when local
hooks (`docs_guard`, `qa_scenario_guard`) didn't run.

**Bypass affordance.** For a checklist item that legitimately
doesn't apply to your change, replace `[ ]` with `[N/A]`. That
mirrors the project's `[docs-not-needed: <reason>]` convention but
at line granularity, and it's reviewer-visible — the box doesn't
silently disappear, it announces that you considered the rule and
made a deliberate call. Don't delete checklist lines; leaving the
`[N/A]` in keeps the template legible from PR to PR.

Enforcement today is honor-system. A CI gate that fails on un-ticked
boxes is a possible follow-up if drift recurs — see
[#271](https://github.com/jackal998/photo-manager/issues/271) for
the design discussion.

CLI users opening a PR via `gh pr create` can pipe the template
straight into the body:

```
gh pr create --body-file .github/PULL_REQUEST_TEMPLATE.md
```

The web UI does this for you automatically.

## Adding a UI string

**Every user-facing string in the app lives in `translations/<locale>.yml`,
not in a Python literal.** This is the single most common thing to get
right (or wrong) when touching the views layer.

### When you add or change a UI string

1. **Add the English value to [`translations/en.yml`](translations/en.yml)**
   under the appropriate namespace. Keys are dotted (`menu.file.scan_sources`),
   lowercase, snake_case. Conventions live in [`docs/i18n.md`](docs/i18n.md).
2. **Look up the string via `t("your.key")`** in the code:
   ```python
   from infrastructure.i18n import t

   button = QPushButton(t("scan_dialog.start_button"))
   ```
3. **Add the same key to every other `translations/*.yml`** with a
   translated value. If you don't speak the target language, leave the
   English value in place — the catalog still loads, and missing-key
   fallback hands the user English instead of breaking the screen. A
   translator catches up later.
4. **Run the tests.** The
   [`test_zh_tw_has_every_key_present_in_english`](tests/test_i18n.py)
   parity test fails the build if a key is in `en.yml` but missing from
   `zh_TW.yml`. That's deliberate — it catches half-translated PRs at
   review time.

### When you rename a UI string

1. Update `translations/en.yml` and every `translations/*.yml`.
2. Grep `qa/` for the old English value — drivers and constants in
   [`qa/scenarios/_uia.py`](qa/scenarios/_uia.py) couple to the
   English UIA accessible names. Update every match.
3. The
   [`test_uia_label_coupling`](tests/test_uia_label_coupling.py) test
   scans both `app/*.py` and `translations/*.yml` for every UIA
   constant — a stale constant fails CI.

### Don't translate

- Logger output and exception messages (developer-facing).
- EXIF field names, manifest schema, internal classifier action names
  (`EXACT`, `KEEP`, `MOVE`, `UNDATED`, `REVIEW_DUPLICATE`) — format
  internals that must round-trip identically across language changes.

The full translator workflow, key-naming rules, and what's deliberately
not translated lives in [`docs/i18n.md`](docs/i18n.md).

## Adding a new locale

Phase 1 ships English (`en`) and Traditional Chinese (`zh_TW`). To add
another language (e.g. Japanese, Simplified Chinese):

1. Copy `translations/en.yml` to `translations/<code>.yml`.
2. Update the `_meta` block:
   ```yaml
   _meta:
     display_name: 日本語
     locale: ja
   ```
3. Translate every value. Keep the keys unchanged.
4. Add a parity test in `tests/test_i18n.py` mirroring
   `test_zh_tw_has_every_key_present_in_english`.
5. Restart the app — the new locale appears automatically in
   `View → Language`.

No Python edits required to ship a locale. The picker walks
`translations/*.yml` at startup.

## Testing rules

The full strategy — three layers, what each catches, per-module
residual risk — is in [`docs/testing.md`](docs/testing.md). Read that
before adding tests. Hard floor for every PR:

- **Layer 1 (`pytest`) must pass.** Runs in CI on every push via
  [`.github/workflows/tests.yml`](.github/workflows/tests.yml).
- **Per-file coverage floor: 70%.** Enforced by
  [`scripts/check_coverage_per_file.py`](scripts/check_coverage_per_file.py),
  run after `pytest` in CI.
- **Global coverage floor: 80%.** Enforced by `pyproject.toml`.
- **No test padding.** A test that exists only to clear a coverage
  gate is metric gaming, not engineering. If you find yourself
  writing a test whose only assertion is "this branch was reached,"
  delete it. Either the branch catches a real bug (test the *real*
  failure mode — a truncated file, a missing optional dep, a
  malformed JSON) or it's dead defense and the right move is a
  comment in source, not a synthesized test.

  Concrete anti-patterns to avoid:

  - Monkeypatching `QStandardItem.setData` to raise so a wrapped
    `except: pass` branch executes.
  - Forcing `_HASH_AVAILABLE = False` to cover the `ImportError`
    fallback when PIL is in fact a hard dependency.
  - Stubbing an `Image` object without `getexif` to cover the
    defensive `if not exif: return None` guard.

### When unit tests can't reach a code path

Some files genuinely can't be unit-tested at layer 1 (require a
running `QApplication`, a real subprocess, real filesystem semantics).
Add the file to `[tool.coverage.run] omit` in `pyproject.toml` with
a one-line comment naming **(a)** why it can't run in unit tests and
**(b)** where it IS covered (a specific qa-explore scenario, an
integration test, manual smoke). Adding to `omit` is a deliberate,
reviewable change — not a per-file slip.

Existing entries in `pyproject.toml` are the reference shape. The
matching row in [`docs/testing.md`](docs/testing.md)'s per-module
table needs updating in the same PR.

### Three triggers, three test homes

1. **Pure logic** → unit test under `tests/`.
2. **Touches a boundary** (subprocess, third-party lib, real
   filesystem) → unit test for our side, layer-3 (`qa-explore`) for
   the happy path. Layer-2 spot-tests are reactive — only when a
   specific boundary bug surfaces.
3. **User-facing flow** (button, dialog, menu) → extend or add a
   `qa/scenarios/sNN_*.py` driver and register it in
   `qa/scenarios/_batch.py:ALL_SCENARIOS` and
   `qa/scenarios/_config.py:SCENARIO_SOURCES`. The full authoring
   guide (naming, helpers in `_uia.py`, fixture conventions,
   anti-patterns) lives in
   [`qa/scenarios/AUTHORING.md`](qa/scenarios/AUTHORING.md).

The QA batch runs separately on PRs touching UI/scanner/QA paths
via [`.github/workflows/qa-batch.yml`](.github/workflows/qa-batch.yml).
It's currently advisory (not a required check) — but a red qa-batch
on a UI PR is a genuine signal, don't ignore it.

## Code style

- **Linters:** Black, isort, Ruff, Pylint. Run via
  `python run_all_linters.py` before pushing.
  **Note: lint is NOT gated on CI today** — only pytest + coverage
  are. That means a lint-only failure won't block your PR, but it
  also means lint regressions slip through if nobody runs the local
  command. Treat the local script as part of "ready to push."
- **Files:** typically 200–400 lines, 800 max. Many small files >
  few large files. Organize by feature/domain, not by file type.
- **Comments:** default to none. Add one when the *why* is non-obvious
  (a hidden constraint, a workaround for a specific bug, behavior
  that would surprise a reader). Don't explain *what* — well-named
  identifiers do that. Don't reference the current task or PR
  number — that belongs in the commit message.
- **Error handling at boundaries.** Validate user input, external API
  output, and file content. Don't add error handling for impossible
  scenarios.

### Project conventions

A few baseline expectations enforced by the linters and code review,
worth knowing up front so your first PR doesn't bounce on style:

- **`from __future__ import annotations` at the top of every Python
  file.** PEP 563 forward references — already present in every
  module; keep the convention.
- **`from loguru import logger`, not `import logging`.** Loguru is
  the project's logger. Sink configuration lives in
  [`infrastructure/logging.py`](infrastructure/logging.py); logs
  rotate to `%LOCALAPPDATA%/PhotoManager/logs/`. Don't introduce
  stdlib `logging`.
- **Import order:** stdlib → third-party → first-party
  (`app`, `core`, `infrastructure`). Enforced by isort + Ruff;
  `pyproject.toml` has the `known_first_party` config.
- **Settings stored under `infrastructure/settings.py:JsonSettings`**
  — dotted-key access (`settings.get("ui.locale", "en")`). When you
  add a new persistable preference, follow this pattern and update
  [`settings.json.example`](settings.json.example).

## Commit and PR conventions

- **Branch off `master`.** No commits directly on `master`.
- **Imperative commit messages**, under 72 chars. Describe the *why*,
  not the *what*. Example: `add zh_TW locale for scan dialog` —
  not `update scan_dialog.py`.
- **One concern per PR.** A bug fix doesn't need surrounding cleanup;
  a feature doesn't need an unrelated refactor. If you spot something
  worth fixing while you're in the area, mention it — don't bundle.
- **No `--no-verify`.** The repo doesn't ship git hooks today, but if
  one lands later and fails on your commit, fix the underlying issue
  rather than bypassing.
- **CI must be green** before merge. The required checks today are
  pytest + per-file coverage (`tests.yml`). Lint is local-only — see
  the Code style section. If the qa-batch workflow runs on your PR
  and goes red, treat it as a real signal even though it's advisory.

## Adding a news fragment for your PR

Every PR adds a one-line entry to the next release's changelog by
dropping a small file into [`news/`](news/). The
[`news-gate`](.github/workflows/news-gate.yml) CI workflow fails any
PR that doesn't, so this is mechanical — but the rules below keep the
generated [`CHANGELOG.md`](CHANGELOG.md) readable.

### Filename — `news/<PR-number>.<type>`

| Type      | Use it for                                                      |
|-----------|-----------------------------------------------------------------|
| `feature` | User-visible new behaviour (new menu item, dialog, scoring)     |
| `bugfix`  | Fixes a wrong behaviour a user could hit                        |
| `doc`     | Documentation only (README, CONTRIBUTING, docs/, comments)      |
| `removal` | Removed feature, removed dependency, breaking schema change     |
| `misc`    | Refactor, CI, tooling, anything else without user-visible diff  |

The filename is the **PR number** (not the issue number). If you don't
know the PR number yet, open the PR first with a placeholder and
update the filename in a follow-up commit before merge.

### Content — one line, present-tense imperative

```text
Add Execute Action preview pane (#165).
```

- One line, no trailing prose. The line is concatenated with bullets
  in the rendered changelog.
- Present tense, imperative — "Add foo", not "Added foo" or "Adds foo".
- If the PR closes a tracked issue, end with `(#<issue>)`. Towncrier's
  `issue_format` renders the link automatically.

### Bypass — `[skip-news: <reason>]`

A PR that genuinely has no entry worth recording (e.g. a typo fix in
a code comment, a transitive-dep bump with no behaviour change) can
skip the gate by including the literal token `[skip-news: <reason>]`
in the PR title or body. The reason is for the reviewer — keep it
specific (`[skip-news: comment typo]`, not `[skip-news: trivial]`).

This is the same convention as `[docs-not-needed]` for the docs-guard
hook.

### Release-time mechanics (you usually don't run this yourself)

At release time, the maintainer runs:

```bash
python -m towncrier build --version <next-version> --yes
```

This consumes every fragment file in `news/`, prepends a versioned
section to [`CHANGELOG.md`](CHANGELOG.md), deletes the consumed files,
and stages everything. You don't run this in your feature PR — only
fragment authoring.

To preview what the next release section would look like without
modifying any files:

```bash
python -m towncrier build --draft
```

## Where to ask

- Behavior questions about photo-manager itself: open an issue.
- Translator workflow / new-locale logistics:
  [`docs/i18n.md`](docs/i18n.md).
- "What does each test layer cover?" / "Where do I add this test?":
  [`docs/testing.md`](docs/testing.md).
- Standing rules for AI-assisted work (security gates that an agent
  must confirm before acting, autonomy boundaries, mid-task pause
  protocol): [`CLAUDE.md`](CLAUDE.md). Most of those rules are
  agent-facing; the always-on subset (no secrets in logs, treat
  third-party input as untrusted, flag CVEs, don't bypass tool
  errors with `--force` / `--no-verify`) applies to human
  contributors too.
