# Localization (i18n)

> **If you're adding or changing a user-facing string:** add the
> English value to [`translations/en.yml`](../translations/en.yml),
> look it up via `t("your.key")`, and mirror the key into every other
> `translations/*.yml`. The
> [`test_zh_tw_has_every_key_present_in_english`](../tests/test_i18n.py)
> parity test fails CI if you skip step 3 — that's the one that catches
> half-translated PRs at review time.
>
> Quickstart for new contributors lives in
> [`CONTRIBUTING.md`](../CONTRIBUTING.md). The rest of this file is
> the deep-dive reference.

Phase 1 ships English (`en`) and Traditional Chinese (`zh_TW`).
Adding more locales is a YAML-only change — no Python edits required.

## Where strings live

```
translations/
├── en.yml      # canonical reference; every UI string in the app
├── zh_TW.yml   # 繁體中文 (Taiwan)
└── README.md
```

`en.yml` is the **single source of truth** for what strings the app
contains. Every other locale file mirrors its key structure exactly.

## How a string gets to the screen

1. A view module imports `t` from `infrastructure.i18n`.
2. It calls `t("some.dotted.key")` (optionally with format kwargs).
3. `t` resolves through the active locale → English fallback → key
   string itself, in that order.

```python
from infrastructure.i18n import t

button = QPushButton(t("scan_dialog.start_button"))
status.showMessage(t("status.manifest_loaded_pairs", pairs=p, files=f))
```

The active locale is picked at startup from `settings.json` →
`ui.locale` (default `"en"`). Changing it requires app restart —
Qt widgets cache their text, and re-translating live would mean
rebuilding every menu, dialog, and label.

## Adding a new locale

1. Copy `translations/en.yml` to `translations/<code>.yml` (e.g.
   `ja.yml` for Japanese, `zh_CN.yml` for Simplified Chinese).
2. Update the `_meta` block:
   ```yaml
   _meta:
     display_name: 日本語   # shown in View → Language picker
     locale: ja
   ```
3. Translate every value. Keep the keys identical — they're how the
   code looks strings up.
4. Restart the app. The new locale appears automatically in
   `View → Language` (the picker walks `translations/*.yml` at
   startup; no registration required).

The unit test
[`test_zh_tw_has_every_key_present_in_english`](../tests/test_i18n.py)
checks every shipped locale against the English baseline. Add a
`test_<code>_has_every_key_present_in_english` for each new locale to
catch missed keys at PR time.

## Key naming conventions

```
<area>.<subarea>.<purpose>
```

Predictable, scannable, and easy to search. Examples:

- `menu.file.scan_sources` — a menu action
- `scan_dialog.title` — a dialog title
- `column.file_name` — a tree column header
- `status.verb_removed` — a status-bar verb

Rules:

- **Mnemonic `&` stays in the value, not the key.** Translators may
  pick a different letter for their language (e.g. `&File` vs.
  `檔案(&F)`).
- **Format placeholders are named:** `{n}`, `{path}`, `{error}`.
  Never positional `{}` or `{0}` — they're brittle when a translator
  reorders sentence parts.
- **Singular and plural are separate keys.** Languages without
  grammatical plurals (Chinese, Japanese, Korean) use the same value
  for both — but having both keys means we never auto-derive
  `noun + "s"`, which would mangle non-English strings.

## Format placeholder safety

If a translator writes `{wrong_name}` and the code passes `name=...`,
`str.format` raises `KeyError`. The `Translator.t` method catches that
and returns the unformatted string — the UI shows literal
`Hello {wrong_name}` instead of crashing. That's a deliberate choice:
a visible glitch is much easier to spot during review than a silent
exception that propagates somewhere awkward.

## Restart-required UX

Switching locale via `View → Language → <choice>`:

1. Persists `ui.locale` to `settings.json`.
2. Updates the checkable indicator on the language items.
3. Shows a `QMessageBox.information` (titled "Restart Required") in
   the **current** locale — the user can read it before having
   restarted.
4. Leaves the app running. The next launch picks up the new locale.

## What's NOT translated

- **Logger output / exception messages.** Developer-facing.
- **EXIF field names** (`DateTimeOriginal`, etc.). Format-internal.
- **Manifest schema fields** stored in `.sqlite`. Stable across
  language changes by design — a manifest written under English must
  read identically when reopened under Chinese.
- **Internal classifier action names** (`EXACT`, `KEEP`, `MOVE`,
  `UNDATED`, `REVIEW_DUPLICATE`). The display label
  ("Ref" / "100%" / percentage) IS translated; the underlying string
  stored in records is not.
- **The internal English field names** in
  [`select_dialog.py`](../app/views/dialogs/select_dialog.py) (used
  as regex-matcher dispatch keys). The combo box displays translated
  labels via `_FIELD_LABEL_KEYS` but emits the English name back.

## Layer-3 verification

QA scenario [`s22_language_switch.py`](../qa/scenarios/s22_language_switch.py)
drives `View → Language → 繁體中文`, confirms the restart-required
prompt, and verifies the persistence to `settings.json`. It restores
`ui.locale=en` before exiting — without that cleanup, every following
scenario would launch in Chinese and break on label assertions.

The "actually rendered in Chinese after restart" check is currently
manual: edit `settings.json` to `"ui": {"locale": "zh_TW"}`, restart
the app, eyeball the menus and dialogs.
